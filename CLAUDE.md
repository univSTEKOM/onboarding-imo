# CLAUDE.md

Operating instructions for Claude (and any AI agent) working in the **Nestplate** monorepo.
Read this before making changes. It is the top-level rulebook; the per-side guides go deeper.

## What this repo is

A full-stack starter template:

- **`backend/`** — NestJS 11 API running on **Bun**, persisting to PostgreSQL via TypeORM.
  JWT + Google OAuth auth, RBAC, a generic `BaseService<T>` CRUD layer, S3 media, Mailgun
  email, Socket.IO notifications, caching, rate limiting, health checks.
- **`frontend/`** — React 19 SPA on Vite + TanStack Router, HeroUI + Tailwind 4, Zustand +
  react-query state, PWA support.

The two sides are independent Bun packages. There is **no root package manager workspace** —
`cd` into the side you're working on and run its scripts there.

## The two mental models (never violate)

**Backend:** Controllers are thin (HTTP only) · Services extend `BaseService<T>` and **emit
events** for side-effects · Guards authorize · DTOs validate · Entities define the schema ·
**nothing is hard-deleted** (every entity has `@DeleteDateColumn`).

**Frontend:** Zustand = client state · react-query = server state · URL search params = table
state · react-hook-form = form fields. No React Context for app state; cookies are the auth
source of truth.

## Hard rules (these break the build or silently rot the system)

Backend:
- **Never hard-delete.** Use `BaseService.remove()` (soft delete).
- **Extend `BaseService<T>`** for CRUD — never re-implement raw repository CRUD.
- **Decouple side-effects through `EventEmitter2`.** Never inject `NotificationsGateway` or
  `MailService` into a feature service; emit an event and let `NotificationsListener` /
  `AuthService` react.
- **Entities must end in `.entity.ts`** — the TypeORM glob in `app.module.ts` discovers them
  by that name. Renaming the suffix makes the table vanish.
- **Never `bun dist/main.js` after a bare `nest build`.** Use `bun run build` (it runs
  `tsc-alias`) or `@/` imports fail in production.
- **Don't hand-edit `swagger.json`** — regenerate with `bun run spec:generate`.
- **Don't wrap success responses** — return the payload; `AllExceptionsFilter` shapes errors.

Frontend:
- **Never edit `src/routeTree.gen.ts`** — generated by the TanStack Router Vite plugin.
- **Never cache server data in Zustand**, and never put pagination/search/sort in `useState`
  — use react-query and URL params (`useDataTable`) respectively.
- **Never reintroduce React Context / `src/lib/providers/`** for app state — it was removed in
  favor of Zustand stores in `src/lib/stores/`.
- **HeroUI is v2.8** — use `Popover` / `PopoverTrigger` / `PopoverContent`, never v3
  dot-notation (`Popover.Content`). Use `Popover` (not `Tooltip`) for hover-revealed content.
- **Don't make `src/lib/api.ts` read the auth store** — cookies are the source of truth.
- **Don't navigate from inside a store** — return a result and navigate in the caller.

## Adding a permission — keep it in sync across all four places

This is the single most error-prone task. When you add a permission string:

1. `backend/src/database/seeder/seeder.service.ts` → `permissionsData` array (so it exists in
   the DB; then run `bun run seed`).
2. `backend/src/auth/permissions.type.ts` → the `PermissionType` union (or `@Permissions()`
   won't type-check).
3. `frontend/src/lib/constants/sidebar.tsx` → `MENU_PERMISSIONS` (only if it's a `menu.*`).
4. `frontend/src/lib/constants/sidebar.tsx` → `sidebarData` (only if it needs a nav item).

A `menu.*` permission missing from the DB means **no user will ever see that nav item.**

## Adding a feature — follow the recipes, don't improvise

- **Backend CRUD module:** entity → DTOs → service (`extends BaseService`) → controller →
  module → register in `app.module.ts` → seeder + `permissions.type.ts`. Full skeleton:
  [`backend/docs/ADDING_FEATURES.md`](./backend/docs/ADDING_FEATURES.md).
- **Frontend CRUD page:** route file → `columns.tsx` → `modal.tsx` → page hook → service →
  schema → sidebar entry → backend seeder. Full skeleton:
  [`frontend/docs/ADDING_FEATURES.md`](./frontend/docs/ADDING_FEATURES.md).

A backend resource and its frontend page are usually added together, and both touch the
permission set — do all of it, not half.

## Verifying changes (definition of done)

Run these from the side(s) you touched **before** declaring a task complete:

**Backend** (`cd backend`):
```bash
bun run lint          # eslint --fix
bun run test          # unit + e2e
bun run build         # confirms zero TS errors AND tsc-alias works
bun run spec:generate # only if you changed routes or DTOs
```

**Frontend** (`cd frontend`):
```bash
bun run lint          # eslint
bun run build         # vite build + tsc (type-check) — must be clean
# For UI changes, click through the flow with `bun run dev`.
```

If a check fails, say so with the output — do not report success on a red build.

## Conventions

- **Imports:** backend uses the `@/` alias (→ `src/`); frontend uses `@/` (→ `src/`) too.
- **Filenames:** backend `*.controller.ts` / `*.service.ts` / `*.entity.ts` etc.; frontend
  **kebab-case**, hooks `use-*.ts`, stores `*.store.ts` exporting `use*Store`.
- **Arrays (frontend):** write `Array<T>` — eslint `array-type` enforces it.
- **DTOs (backend):** `BaseXDto` → `CreateXDto` (adds `@IsUnique`) → `UpdateXDto =
  PartialType(...)`. `@ApiProperty()` on schema fields, `@Exclude()` on secrets.
- **Mutations (frontend):** always `useAppMutation` with `invalidateKeys` + `successMessage`.
- **Pagination/tables (frontend):** always `useDataTable(Route)`; truncating columns use
  `truncate: true` + `maxWidth`.
- Match the surrounding code's style, comment density, and idioms.

## Environment & secrets

- Backend env is validated by a Zod schema in `backend/src/config/env.ts` and **throws on
  startup** if a required var is missing. When you add an env var, add it there *and* to the
  table in `backend/README.md`.
- Never commit real secrets. `.env` files are git-ignored; `backend/.env.example` is the
  template.

## Working agreement

- Confirm before destructive or hard-to-reverse actions (dropping the DB, force-push, deleting
  files you didn't create). `bun run migrate:fresh` **drops the schema** — don't run it
  casually.
- Don't add a dependency when an installed one already does the job.
- Commit/push only when asked. If on `main`, branch first. Git author is `andikadevs`.
- Keep these docs honest: if you change behavior the docs describe, update the doc in the same
  change.
